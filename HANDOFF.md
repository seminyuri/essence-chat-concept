# Передача бэкенд-разработчику — «Суть ✕ Чат»

> Это **дизайн- и поведенческий референс** (не прод-фронт): чистый JS без фреймворка,
> данные-плейсхолдеры в памяти. Он задаёт **как выглядит и как ведёт себя** мессенджер.
> Реальный фронт строится на стеке платформы (Rails/Hotwire), бэкенд — по контрактам ниже.
> Telegram — второй референс поведения (раскладка, переключения, жесты).

Файлы: `index.html` (каркас + SVG-спрайт иконок), `assets/app.js` (вся логика/рендер),
`assets/chat.css` (стили модуля; токены — в `assets/essence.css`, `shell.css`).

## 2. Три аудитории — как смоделированы

1. **Внутренние кросс-орг чаты.** Орги (`ORGS`) — изолированные воркспейсы; чаты силосятся по орге в `CHATS[org]`; глобальный переключатель + «Все» (флэт по всем оргам с тегом орги на строке). Филиалы Суши Мамы смоделированы как **папки + групповые чаты** (не вложенные суб-воркспейсы — см. §5).
2. **Специалист → клиент по ссылке.** Инвайт-модалка с тумблером «запрашивать дату/время/место рождения» (для клиента — да, для внешнего — можно выключить). Лендинг: OAuth (Telegram/VK/MAX) → шаг натал-данных (если требуется) → «в чате». Чат клиента: риббон + натал-панель + **«Открыть карту»** (SVG-колесо).
3. **Внешние/поставщики через MAX/VK, без натала.** Тип `external`, риббон «натал-данные не запрашиваются», в профиле — канал входа.

## 3. Сущности и поля (контракты, выведенные из UI)

Бэкенд должен определить схему; ниже — что подразумевает интерфейс.

**Organization** `{ id, name, logo, accentTokens, memberRole(userId) }` — воркспейс. Роль пользователя в орге: `owner|admin|member` (сейчас риббон показывает «Владелец/Администратор/Личное»).

**User** `{ id, name, avatar, presence(online|recently|…), kind(client|specialist|external|bot|staff), via(MAX|VK|TG для external), natal|null }`.

**Natal** `{ date, time, place }` — сейчас **свободный русский текст без гео/таймзоны**. Для реального построения карты нужны структурированные `{ date(ISO), time, lat, lon, tz }`. Колесо в модалке сейчас **захардкожено** — реальную карту считает астро-движок платформы.

**Chat** `{ id, orgId, type(dm|group|channel|client|external|saved), title, memberIds[], folder, pinnedMsgIds[], muted, archived, subscribers(channel), last }`. Тип определяет: композер (канал=broadcast), риббон (client/external), натал-панель (client).

**Message** `{ id (стабильный, глобально-уникальный!), chatId, fromUserId, ts, text, replyToMsgId, media, file, voice{wave,dur,transcript}, poll{q,options[{t,v}],votedBy}, sticker, link, reactions[{emoji, byUserIds[]}], readByUserIds[], edited, forwardedFrom }`.
> ⚠️ В прототипе id уникальны только **внутри треда** (`m0,m1…`) — бэкенд обязан выдавать **глобально-уникальные** id. Пины и разделитель непрочитанного в прототипе я довёл до устойчивости при удалении, но правильнее хранить их **по message-id**, а не по индексу.

**Invite** — сейчас кодируется только префиксом ссылки (`c-`/`x-`) и `?req=`. Бэкенду нужна сущность-токен: `{ token, orgId, specialistId, audience(client|external), requireNatal, expiresAt }`, чтобы связать «какой инвайт → какой чат/пользователь».

**Membership / Permissions** — сейчас роли — inline-литералы в рендере. Нужна связь `membership {userId, orgId|chatId, role, permissions}`. 5 групповых прав из `groupAdminModal`: отправлять сообщения / медиа / добавлять участников / закреплять / менять название-фото; + медленный режим (off/30s/1m/5m); + тип группы (приватная/публичная).

## 4. Права и «кто кого видит»

- **Орг-роли:** owner/admin/member (кто может обходить членство).
- **Групповые права:** 5 тумблеров + slow-mode + приватная/публичная (`groupAdminModal`).
- **Приватность (личная):** кто видит телефон/фото/«был в сети»; кто может добавлять в группы/звонить; «Отправлять Прочитано» (`settings → privacy`).
- **Видимость между оргами:** воркспейсы изолированы. Единой матрицы «кто кого видит по оргам» в UI нет — если нужна, это отдельный экран/модель.

## 5. Что заглушки (бэкенд делает по-настоящему)

- **Натальная карта** — колесо и разбор захардкожены; считает астро-движок из структурированных данных рождения.
- **Membership/роли/права** — сейчас литералы; нужна реальная схема членства + capability-маппинг тумблеров.
- **Инвайт-токен** — не персистится; нужна сущность (§3).
- **Филиалы = папки**, не суб-орги. Если филиалам нужна своя изоляция/членство/роли — нужен структурный контейнер.
- **История сообщений** синтетическая, без пагинации/персистентности; всё состояние в памяти.
- **Тосты «— в полной версии»** (добавить участников, запись голоса, bulk-forward, создать оргу, выход из сеанса/орги, очистить историю, назначить админом) — намеренные placeholder-действия.
- **Нативные контролы, чьё значение не читается** (тумблеры настроек, `<select>` приватности, чекбокс «Публичный» канала) — визуальные плейсхолдеры.
- **Время «прочитано»** в «кем прочитано» — синтетика; модели времени прочтения нет.

## 7. Что я НЕ гарантирую

Это прошло два аудита (мёртвые кнопки + логика) и правки, но это **не прод-код и не
QA-сертификация**. Прототип на плейсхолдерах: при живых данных, конкурентности, персистентности
и реальных правах вскроется то, чего в статике не видно. Используй как спеку поведения/дизайна,
а не как код к расширению.
